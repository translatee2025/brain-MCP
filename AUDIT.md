
# Brain MCP — Audit & Ship-Ready Roadmap

## 1. What it is

Brain MCP is a single-process, local-first MCP stdio server that gives Claude Desktop (and ChatGPT desktop) a private, persistent long-term memory. The model writes a clean Markdown note and calls a tool; the server stores it in a local SQLite database at `~/.brain/brain.db` with full-text search (FTS5), tags, folders, and version history. It ships as a one-click `.mcpb` desktop extension, has no account, no telemetry, and is MIT-licensed across roughly six source files (`src/index.ts`, `src/server.ts`, `src/db.ts`, `src/restructure.ts`, `src/export.ts`, `src/ui.ts`) and five runtime dependencies.

The core design thesis is stated plainly in `HANDOFF.md`: **the server is a dumb but well-indexed Markdown store, and Claude is the brain.** All intelligence lives in the model, not the server. The store's job is to persist, index, version, and never lose what it is given. Because an MCP server cannot passively watch a conversation, the headline "remember after about ten pages" behavior is implemented as a model-mediated running buffer (`brain_capture`) that signals when to flush into a note (`brain_flush`), driven by a documented project instruction. Optional server-side cleanup is pluggable across three engines (host model via MCP sampling, any OpenAI-compatible endpoint including local Ollama or LM Studio, or none) and always degrades gracefully to verbatim storage so a note is never lost to an engine failure.

## 2. Feature inventory

**Note lifecycle (core CRUD)**
- `brain_save`: stores a clean Markdown note with title, body, optional folder, tags, and an optional `id` to overwrite an existing note. Returns Created/Updated, the doc id, and which restructure engine ran. (`src/server.ts` -> `restructure()` -> `saveDoc()` transaction in `src/db.ts`.)
- `brain_get`: fetches one note in full by id, with a header line of folder and tags. (`getDoc()` in `src/db.ts`.)
- `brain_list`: lists recent notes newest-first, optionally within a folder, prefixed by a per-folder count summary. Limit 1 to 200, default 50.
- `brain_delete`: soft-deletes by id (dropped from search and listings, row retained). (`deleteDoc()` transaction.)

**Search**
- `brain_search`: FTS5 full-text search over non-deleted notes, bm25-ranked, with highlighted snippets and an optional folder filter. `sanitizeQuery()` strips terms to letters and numbers, lowercases, caps at 16 terms, and quotes each so a stray `:` or `(` cannot throw an FTS syntax error.

**Versioning**
- `brain_versions`: lists a note's version history newest-first (version number, `changedBy`, timestamp).
- `brain_restore`: restores a note to an earlier version, recorded as a new version tagged `rollback:vN`.
- Version retention: keeps the last 20 versions per note (`VERSION_KEEP=20`, `pruneVersions` in `src/db.ts`).

**Capture / auto-save buffer**
- `brain_capture`: appends the latest exchange to a per-session buffer, returns a token estimate, threshold, and `should_flush`.
- `brain_flush`: turns the whole buffer into one note, saves it, and clears the buffer. Threshold is `BRAIN_FLUSH_TOKENS` (default 6000, about ten pages, estimated at four chars per token).

**Restructuring engines**
- Pluggable cleanup before storage via sampling (host model, `server.server.createMessage()`, capped at `maxTokens: 4000`), an OpenAI-compatible endpoint (`viaOpenAI`), or none. Any failure or empty result falls back to verbatim storage (`engine: 'raw'`). (`src/restructure.ts`.)
- `brain_system_check`: inspects RAM/CPU/platform via `node:os` and recommends a right-sized local model tier plus exact Ollama setup steps, instead of blindly downloading gigabytes.

**Export and UI**
- `brain_export`: zip export of folder-structured Markdown with YAML frontmatter, a `_index.md` table of contents, and `_metadata.json`. (`src/export.ts`, `adm-zip`.)
- `brain_open_ui`: starts a localhost-only web viewer/editor (bound `127.0.0.1`, `BRAIN_UI_PORT` default 4319) and opens it. Sidebar of folder-grouped notes with search and download buttons; main pane renders Markdown, edit/view toggle, save, delete, version history with restore. (`src/ui.ts`.)
- Web UI HTTP routes: `GET /`, `GET /api/docs`, `GET /api/search`, `GET /api/doc/:id`, `PUT /api/doc/:id`, `DELETE /api/doc/:id`, `POST /api/doc/:id/restore`, `GET /download/doc/:id`, `GET /download/all`, `GET /download/db`.

**MCP resources and persistence**
- `brain://doc/{id}` resource: exposes every note as a readable MCP resource (list returns up to 200 docs).
- SQLite store: better-sqlite3 with WAL, foreign keys on, `synchronous=NORMAL`; tables `docs`, `tags`/`doc_tags`, `fts` (FTS5 unicode61), `doc_versions`, `capture_buffer`; all multi-table writes in transactions.

**Configuration and packaging**
- Env knobs: `BRAIN_DIR`, `BRAIN_RESTRUCTURE`, `BRAIN_LLM_BASE_URL`/`_API_KEY`/`_MODEL`, `BRAIN_FLUSH_TOKENS`, `BRAIN_UI_PORT`, each mapped to a `user_config` field in `manifest.json` (manifest version 0.4), with `llm_api_key` marked sensitive.

## 3. Strengths

- **Genuinely local-first and privacy-clean.** Zero telemetry, one SQLite file, network egress only via two explicit user-opted paths (MCP sampling or a self-configured OpenAI-compatible endpoint). The `manifest.json` default `restructure_mode='none'` means no traffic at all for the one-click install.
- **Robust graceful degradation.** Every restructure path falls back to verbatim storage, so a note is never lost to an engine outage, a missing key, or an unsupported host.
- **Non-destructive data model.** Soft-delete keeps rows, restore records new versions, and last-20 retention bounds growth.
- **Defensive FTS handling.** `sanitizeQuery` quotes and strips terms and caps at 16, so malformed input cannot throw an FTS5 syntax error. bm25 ranking plus highlighted snippets give useful results.
- **Small, legible, low-dependency surface.** Six source files, five runtime deps, mature primitives (prepared statements, transactions, WAL). Easy to audit and maintain.
- **Pragmatic answer to MCP's lack of a transcript feed.** The capture/flush buffer plus a documented project instruction is an honest, working approximation of auto-save.
- **Good escape hatches.** Full zip export with frontmatter, index, and metadata; raw DB download; a usable localhost editor with version restore. The user is never locked in.
- **Hardware-aware `system_check`.** Recommends a right-sized local model rather than a blind multi-gigabyte download.
- **A committed lockfile and reproducible deps.** `package-lock.json` (lockfileVersion 3) is checked in, contrary to one early reviewer assumption (see rejected findings).

## 4. Security audit (confirmed findings only)

**What was investigated.** The actual `brain-mcp.mcpb` bundle and all `src/` files were inspected. Findings below are only those independently verified at high confidence against the real source and, where applicable, reproduced. Severities reflect the product's real context: a local, single-user, no-account, loopback-only tool whose own `HANDOFF.md` (section 10) already documents that the UI has no auth and that HTML sanitization is "not a security boundary." That documented threat model lowers, but does not erase, these findings: the product's entire promise is "your memory never leaves your Mac," and the web-UI findings directly threaten that promise.

| Severity | File / location | Issue | Impact | Fix |
|---|---|---|---|---|
| **High** | `src/ui.ts` startUi() / handler, lines 130-244 | **No auth, no Origin/Host validation: localhost UI fully exploitable via DNS rebinding.** The server binds `127.0.0.1:4319` with no token, no Origin check, and no Host-header validation; it routes purely on path/method. SOP blocks plain cross-origin JSON reads, but a malicious page re-resolving its host to `127.0.0.1` becomes same-origin, after which every endpoint is readable and writable. | Any website visited while the UI is running (it stays up for the process lifetime after `brain_open_ui` is ever called) can read every note, download the whole `brain.db` and zip, and silently edit or delete notes. Silent, complete exfiltration of the private knowledge base. | Validate the Host header on every request (reject anything not exactly `127.0.0.1:<port>`/`localhost:<port>`); require a per-launch bearer token embedded in the opened URL on every `/api` and `/download` route; validate Origin on all requests; bind a random ephemeral port; add an idle-timeout shutdown. |
| **Medium** | `src/ui.ts` PUT/DELETE lines 169-185, POST restore 188-193 | **State-changing endpoints have no CSRF protection.** No CSRF token, no Origin check, no server-side header gate. `readBody` JSON-parses without enforcing a JSON content-type. PUT/DELETE get incidental preflight protection from a pure cross-origin attacker, but `POST /restore` is a CORS-simple request and is cross-origin-reachable today; under DNS rebinding the full PUT-rewrite/DELETE become reachable. | A malicious page can roll a note back today, and under rebinding can rewrite or delete any note. Because notes are re-fed to the model as `brain://doc/{id}` resources, rewritten notes are a durable prompt-injection / memory-poisoning foothold for future Claude sessions. | Enforce the per-session anti-CSRF token on all mutating requests; reject any mutating request whose Origin/Referer is not the loopback UI; require `Content-Type: application/json`. |
| **Medium** | `src/ui.ts` `/download/db` 225-231, `/download/all` 215-223, `/api/docs` 145-150 | **Raw SQLite DB and full export downloadable over unauthenticated GET.** These are plain GETs (navigable, no preflight). `brain.db` retains soft-deleted notes, full version history, and the capture buffer. | One GET to `/download/db` exfiltrates the entire database, including notes the user believed deleted. (Note: the `/download/all` zip filters `deleted=0`, so the soft-deleted exposure is specific to the raw DB download.) | Gate behind the auth token and Host/Origin validation; consider not serving the raw `.db` over HTTP at all; never serve on a request lacking the loopback Origin and session token. |
| **Medium** | `src/ui.ts` renderMd() 23-29; innerHTML sink line 97 | **Regex HTML sanitizer is bypassable: stored-XSS-adjacent in the UI origin.** `renderMd` runs `marked` (raw HTML passthrough by default) then strips script/iframe/object/embed pairs, `on*=` handlers, and `javascript:` via regex, then injects via `innerHTML`. Bypasses confirmed (unterminated `<script>`, case-mixed tags, `<svg onload>` tag survival). Note content is fully attacker-influenceable via `brain_save` with `restructure:'none'` or the unauthenticated PUT. | A crafted note rendered in the local viewer can inject markup into an origin that can read `/download/db` and all `/api` routes, chaining to a self-contained exfiltration. (Confirmed code-execution payload not found against current `marked` output; the sanitizer is fragile and misleading rather than a proven working XSS.) | Replace the hand-rolled regex with a vetted sanitizer (DOMPurify via jsdom, or sanitize-html), or disable inline HTML in `marked`; add a strict Content-Security-Policy on UI responses. |
| **Low** | `src/export.ts` exportZip() 72-78; `src/server.ts` brain_export 230-238 | **Arbitrary file write via export path.** The optional `path` arg is passed straight to `zip.writeZip(path)` with no extension/allowlist/traversal/overwrite check, and it is model-controlled (prompt-injectable). | File creation/overwrite at an attacker-chosen path as the user. (Impact bounded: `adm-zip` always emits valid PKZIP framing, so this is corruption/DoS of an overwritten file, not a clean injected-payload RCE. Contradicts `PRIVACY.md`, which says exports go to `~/.brain/exports/` only.) | Default to `BRAIN_DIR/exports`; if a path is allowed, require it inside an allowlisted root, enforce `.zip`, reject `..`, refuse to overwrite without confirmation. |
| **Low** | `src/ui.ts` catch block 235-238 | **Generic 500 returns `String(e)` to the client.** Leaks error name and message (SQLite library/SQL-syntax detail, doc ids). Not stack traces or absolute paths in this handler, and only cross-origin-readable under rebinding. | Minor reconnaissance that compounds the rebinding attack. | Return a generic message to the client; log detail to stderr only. |
| **Low** | `src/db.ts` 31, 100; `src/export.ts` | **DB and exports unencrypted; soft-deleted notes and full history retained and exfiltratable.** `brain.db` is plaintext SQLite; delete is soft (body retained in `docs` and `doc_versions`); no purge/hard-delete tool; `/download/db` carries it all. | Content the user explicitly deleted persists indefinitely and can be recovered or exfiltrated, violating the deletion expectation; no encryption means file-level access reads everything. (Per-design and documented in `PRIVACY.md` section 8, but a real deletion-expectation gap.) | Add a true hard-delete/purge of doc + versions + FTS + tags; optional encryption at rest; document that soft-delete retains data; lock down the DB-download path. |
| **Low** | `src/db.ts` captureAppend 294-298, saveDoc 202; `src/ui.ts` readBody 40-49 | **Unbounded input (DoS / resource exhaustion).** No length caps; `captureGet` concatenates the whole buffer on every `brain_capture` (quadratic); `readBody` buffers all chunks with no size cap; no rate limiting. (Version history is NOT unbounded: `VERSION_KEEP=20` caps it.) | A large or repeated input can exhaust memory/CPU and degrade or crash the server and bloat `brain.db`. Local single-user, no confidentiality/integrity impact. | Cap note/content length, cap per-session buffer size, add a request-body size limit in `readBody`, add basic per-connection limits. |

**Investigated and rejected (for trustworthiness).** Several plausible-sounding findings were investigated and confirmed **not** real for this codebase:
- **Zip-slip / path traversal in export folder names** — rejected. The pinned `adm-zip` 0.5.17 normalizes entry names on write (`zipnamefix`), stripping `..` before the entry exists, verified by round-trip. The traversal-to-RCE-on-extract claim does not hold.
- **SSRF via `BRAIN_LLM_BASE_URL`** — rejected. The base URL is sourced only from `user_config` (the local operator), no tool accepts a URL, and the model can only choose the restructure mode, not the endpoint. No attacker-controllable target on a single-user local tool.
- **Capture-buffer `MAX(seq)+1` UNIQUE-violation race** — rejected. WAL serializes writers and the subquery+INSERT is one atomic statement; the real (and different) multi-host issue would be a retryable `SQLITE_BUSY`, not a duplicate-key violation.
- **No `busy_timeout` causes immediate write failures** — rejected. better-sqlite3 sets `busy_timeout=5000ms` by default, so a contended writer waits, contrary to the finding's premise.
- **API key leakage in `restructure.ts`** — rejected. The key is flagged sensitive, never logged anywhere in `src/`, and errors are swallowed. The cited location contains no defect.
- **Unsigned bundle / no lockfile supply-chain risk** — rejected as written. `package-lock.json` is committed and tracked; the residual (unsigned `.mcpb`) is a distribution-hygiene note, not a code vulnerability. (Signing is still tracked as a Directory-readiness item in section 9.)
- **FTS sanitizer hardening** — informational only; the sanitizer already neutralizes FTS5 syntax injection and classic SQL injection (bound parameter).

## 6. Gap analysis (vs competitors and power-user expectations)

The 2026 memory-MCP field splits into three camps: knowledge-graph servers (the official MCP memory server), vector/managed memory (mem0/OpenMemory, supermemory, Zep+Graphiti, Letta, Mimir, Cipher), and Markdown-file-first stores (basic-memory, and the git-as-memory crowd: DiffMem, Memoir, Memoria, Letta Context Repositories). Brain MCP sits squarely in the most crowded, fastest-converging camp.

**Capabilities the field treats as table stakes that Brain lacks:**
- **Semantic / vector retrieval.** Search is pure FTS5 bm25, so recall fails the moment the user paraphrases ("that thing about my landlord" misses a note titled "Lease dispute"). Every serious competitor (mem0/OpenMemory FastEmbed, Mimir's FTS5+vector hybrid, Zep) ships hybrid retrieval. This is the single biggest functional gap.
- **Automatic / proactive recall.** Both capture and retrieval are 100% load-bearing on a pasted project instruction; `brain://doc` only lists notes, it does not retrieve relevant ones. Competitors (claude-mem auto-inject, OpenMemory auto-recall) make this automatic. If the user skips the instruction, the headline feature silently does nothing.
- **Secret redaction and encryption at rest.** For a privacy-positioned tool, these are the two most credibility-damaging gaps: notes (and the `llm_api_key` in `user_config`) are stored verbatim in plaintext SQLite, and the docs themselves warn against pasting credentials. Peers (Mimir AES-256-GCM, SQLCipher) treat encryption as table stakes.
- **Temporal fact-validity and contradiction handling.** No created/updated/superseded model and no near-duplicate detection; saving a contradicting fact silently creates a conflicting twin, the failure mode that degrades long-term memory. Zep+Graphiti leads here.
- **Cross-platform install.** `manifest.json` is darwin-only, `brain_open_ui` only handles darwin/win32 (no Linux `xdg-open`), and the native module breaks cross-ABI (see section 5). Mimir's single-binary distribution is the bar.
- **Editable-plaintext substrate (Obsidian/vault).** Notes live only in an opaque `brain.db`; the only escape is a manual zip. The local-first audience strongly prefers Markdown files they own, which basic-memory and Obsidian-AI tooling are built on.
- **Import, auto-tagging, multi-device sync, observability, benchmarks.** No `brain_import`, no auto-organization, no sync, no logging or stats tool anywhere in the codebase, and no published LoCoMo/LongMemEval numbers, which reads as unproven next to competitors who all cite them.

**Where Brain meets or exceeds the field:** human-readable per-note Markdown versus the official server's opaque JSONL; a built-in browse/edit/version web UI and first-class zip export that basic-memory gates behind its cloud tier; and a genuinely zero-infra, no-Docker, no-vector-DB, no-account local stack.

## 7. Competitive positioning

**Where Brain wins:**
- **Ownership and privacy as the lead, not retrieval sophistication.** "Your AI memory is human files you own, can read, edit, diff, and export, private by default, while Claude does the thinking." No cloud, no Docker, no vector DB, no account.
- **The bundle is the moat.** Local Markdown + SQLite index + version history + a real local web UI + zip export + privacy-by-default rarely co-exist in one tool. basic-memory (the closest competitor) gates web UI and version history behind cloud; the official server and mem0's local dashboard do not match the combination.
- **No query engine to attack.** mem0 shipped multiple CVEs and Graphiti shipped a Cypher-injection CVE via prompt injection in 2026. Brain's "no server-side query engine, just files on disk" model is a real, marketable security advantage for the privacy-conscious.

**Where it must catch up:**
- **Semantic recall** is the one capability the field expects present before recommending a memory tool as standard.
- **Version history alone is no longer a differentiator** — the git-as-memory crowd (DiffMem, Memoir, Memoria, Letta Context Repositories) already owns the versioned, explainable, rewindable narrative. Brain must win on privacy + Claude-native MCP packaging + editable UX + zip export framed together.
- **Pick a lane and hold it.** Stay a passive, model-agnostic, single-user local store. Do not chase Letta's agent-runtime niche or the team-shared cloud niche. Optionally add basic-memory-style Obsidian interop to ride that mindshare.

## 8. Roadmap to ship and to "standard tool"

The strategy is a two-track sprint. Track 1 (eligibility) must land before submission. Track 2 (love and standard-worthiness) is what earns a feature.

### Tier A — Must-fix before submitting to the Directory (security + correctness blockers)

These have a forced order; nothing else matters until Brain can be submitted and installed.

1. **Tool annotations + non-instructional descriptions** (`registerTool` migration, `src/server.ts`). Migrate all 12 tools from legacy `server.tool()` to `registerTool` with a `title` and an `annotations` object: `readOnlyHint` on search/get/list/versions/system_check; `destructiveHint` on save/delete/restore/flush/export. Rewrite descriptions to plain what/when, moving model-instructing prose into the README. **Why it wins:** missing `title`/`readOnlyHint`/`destructiveHint` is the single explicit, listed Directory rejection reason; for the user it produces correct confirm gates so a note is never lost to a surprise overwrite. **Effort: S. Impact: high.**
2. **Cross-platform single-bundle install** (kill the native-module load crash; section 5 high finding). Migrate the store to `node:sqlite` (stable, ships inside Claude Desktop) or WASM SQLite to remove the native compile step; add win32/linux to `manifest.platforms` and an `xdg-open` fallback. **Why it wins:** the tool actually loads on the first try on Mac, Windows, and Linux; reviewers explicitly test Windows + macOS, and a server that fails to load is an instant fail. **Effort: M. Impact: high.**
3. **Lock down the localhost UI** (`src/ui.ts`; section 4 high/medium findings). One pass: per-launch session token required on every `/api` and `/download` route; reject non-loopback Host and non-loopback Origin/Referer (kills DNS rebinding); require the token on all PUT/DELETE/POST; replace the regex stripper with an allowlist sanitizer; add an `EADDRINUSE` `error` handler so a busy port does not crash the MCP process. **Why it wins:** closes the confirmed data-exfil path a reviewer will hit in a 10-minute test; turns "wide-open localhost" into "this is the safe one." **Effort: M. Impact: high.**
4. **Reviewer-ready submission package.** Sign the `.mcpb` (so `mcpb verify` passes); add 3 to 5 screenshots (>=1000px: note list, rendered note, search results, version restore) referenced from `manifest.screenshots`; add `brain_import` (the missing inverse of `export.ts`, with size caps, path-traversal guard, dry-run count) plus a seeded sample vault that imports in one call; write 3 working demo prompts. Also fix the truncation/durability data-loss footguns (Tier A overlaps section 5: chunk large flushes, clear the buffer only after a confirmed save). **Why it wins:** a populated test account, screenshots, signed bundle, and 3 prompts are the literal review checklist; takes Brain from ~70% to acceptable. **Effort: M. Impact: high.**

*Also fold into v1 because it is cheap and load-bearing:* **secret-redaction firewall** (S) — see below; it gates safe export and is the cheapest privacy-credibility win.

### Tier B — Hero features that make it standard-tool-worthy

5. **Bundled fully-offline hybrid semantic search (FTS5 + on-device e5 vectors, fused rank).** Add a `sqlite-vec` table alongside FTS and embed notes and queries with a small bundled multilingual e5 embedder, reusing the exact proven pattern from this user's Dobee project (`resources/embedder/`, fetch-on-install, asarUnpack, 768-dim, query:/passage: prefixes, fully offline). Fuse bm25 + cosine via reciprocal-rank-fusion; degrade to FTS-only if absent. **Why it wins (users):** finds what you meant, not what you typed, months later, in any language. **Why it wins (Anthropic):** closes the table-stakes capability gap while staying the only privacy-clean, zero-egress option. **Effort: L. Impact: transformative.** *Build this first in Track 2; recall, dedup, and graph edges all reuse the vectors.*
6. **Zero-instruction auto-recall: `brain_recall(question, k)` + bundled MCP Prompts.** One tool that sanitizes, runs hybrid search, dedups, and returns top-k citation-tagged snippets; paired with bundled Prompts (`/recall`, `/remember`, `/memory-status`) that auto-register and surface as slash commands. **Why it wins (users):** the magic moment works out of the box, no pasted instruction to forget. **Why it wins (Anthropic):** "just works on install" is what gets featured, and it is a flagship reference example for the MCP Prompts primitive Anthropic is pushing. **Effort: M. Impact: transformative.**
7. **Invisible secret-redaction firewall (redact-before-write).** A deterministic pass inside `brain_save`/`brain_flush` before anything touches SQLite: AWS/GCP/Azure keys, GitHub/Stripe/Slack tokens, JWTs, PEM/SSH blocks, bearer headers, `KEY=secret` lines, high-entropy strings, each replaced with a typed placeholder plus a salted fingerprint, returning a redaction count (never values). **Why it wins (users):** paste a stack trace or `.env` safely by construction. **Why it wins (Anthropic):** directly satisfies the "collect only what's necessary / no sensitive conversation data" rule and is impossible for opaque-vector competitors to match. **Effort: S. Impact: high.**
8. **Optional plaintext-Markdown vault mirror + round-trip import.** Opt-in mirror of each saved note as a frontmatter `.md` into a chosen vault folder (redaction runs before the mirror write), keeping encrypted SQLite as the authoritative index; make export+import a true round-trip. **Why it wins (users):** "my memory is plain files I own," and it solves multi-device sync for free via the user's own iCloud/Syncthing. **Why it wins (Anthropic):** data-ownership/no-lock-in is the privacy lane Anthropic can publicly endorse, and it is the wedge against basic-memory. **Effort: M. Impact: high.**

## 10. Bottom line

Brain MCP is an honest, legible, genuinely local-first memory server whose "dumb well-indexed Markdown store, Claude is the brain" thesis is both its charm and its current ceiling: the bundle of local SQLite + FTS5 + version history + a real web UI + zip export + privacy-by-default is rare and defensible, but almost every individual pillar now has a stronger incumbent, and the field's table stakes (semantic recall, encryption at rest, secret redaction, automatic recall, cross-platform install) are exactly where it looks dated. Today it is roughly 70% Directory-ready and, critically, not shippable as-is: it carries one high-severity packaging crash that breaks the headline one-click install for Intel and ABI-mismatched users, a high-severity wide-open localhost UI that defeats its own "never leaves your Mac" promise via DNS rebinding, and the single explicit hard-reject of missing tool annotations. Close those plus secret redaction in a focused v1, then build the bundled offline embedder, zero-instruction `brain_recall` with MCP Prompts, and an opt-in plaintext vault mirror, and Brain MCP goes from a clean local note box to the cleanest, safest, most obviously-useful, and most recommendable privacy-first memory tool in its category, which is precisely the profile Anthropic features.
